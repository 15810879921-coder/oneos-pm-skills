OneOS AutoVUL Skill · 运作原理说明

分享给测试 / 发版同事 便于按自己项目定制 · 非技术实现手册

一句话原理： 这不是后台自动跑的程序，而是给 AI Agent 的作业说明书。 测试同事输入云效迭代名称，Agent 就按说明书去读该迭代关联需求，整理成 OneOS PC 对外「版本更新日志」成稿。

1. 和 AutoPRD 的分工

Skill 谁常用 干什么
AutoPRD
oneos-autoprd		 产品 写整模块需求说明；定稿时产出「更新内容」写进云效需求
AutoVUL
AutoVUL		 测试 / 发版 按迭代汇总多条需求，生成整包发版对外更新日志
  • AutoPRD = 单条需求怎么写清楚（建单、定稿）
  • AutoVUL = 一整批迭代怎么对外讲清楚（发版公告）

二者都不是文件监听器：没有人在 Cursor 里调用时不会自己跑。

2. Agent 实际怎么走（按迭代）

确认云效项目（默认：统一运营管理平台PC端） → 读取用户给出的「迭代名称」 → 在云效查找迭代（成功 ✅ / 失败 ❌ 立刻反馈） → 列出该迭代关联的全部「需求」 → 优先抽取每条需求描述里的「更新内容」 → 按 AutoVUL 格式分类、同模块合并、润色 → 输出纯文本成稿（无需求号、无预计维护时长）

迭代读取反馈（必须先说清）

结果 Agent 怎么说 你怎么做
找到 1 个迭代 ✅ 已找到迭代「…」，关联需求 N 条 可核对清单；继续出成稿
找不到 ❌ 未找到迭代「…」。请重新输入迭代名称 复制云效迭代标题再发一次
多个相似 ⚠️ 找到多个相似…请选序号 回复序号或完整精确名
未登录 / 无权限 ❌ 无法访问云效… 浏览器登录后重试，或改手动清单

3. 成稿为什么长这样

对齐业务可读发版口径：

  • 仅两类：【新功能】【Bug修复】；同模块合并
  • 每条 = 做了什么 + 对谁有什么用（责任部门自然写入，如「运维部人员可…」）
  • 部门名对照「业务条线说明」；禁止「供某某条线使用」套话
  • 对外正文不写需求号、缺陷号、预计维护分钟（时长人工另发）

4. 和「手工粘清单」的差别

传统发版文案 这套 AutoVUL
测试从各需求复制粘贴，容易漏、口径不一 以迭代为范围，一次拉齐关联需求
写错迭代名只能口头返工 Skill 强制 ✅/❌ 反馈，失败可直接重输迭代名
常带需求号、技术黑话 模板强制产品语言 + 部门价值句

5. 何时使用 / 怎么用（给人看的短版）

何时使用

  • 测试发版前：按云效迭代生成 PC 整包更新日志
  • 工作台「版本更新」弹框 / 对内发版通知需要统一口径
  • 迭代名读失败后，重新输入名称再生成

怎么用

  1. 在 Cursor 终端执行安装命令（或把「发给 AI」文案粘贴给 Agent 代装）。
  2. 对 AI 说：按 $AutoVUL 生成版本更新日志；并给出迭代名称（及可选更新时间）。
  3. 看 Agent 反馈：成功则核对需求清单；失败则重新输入迭代名称。
  4. 确认成稿后对外发布；预计维护时长由人工单独通知。
  5. 云效不可用时：用手动清单模板（input-template.md）兜底。

6. 其他同事怎么安装 / 定制

安装（Cursor）：

npx skills add 15810879921-coder/oneos-pm-skills --skill AutoVUL -a cursor -y

更新：

npx skills update AutoVUL

复制 Skill 文件夹后，主要可改：

  1. SKILL.md 顶部 description — 触发关键词（迭代、发版、AutoVUL）。
  2. 默认项目名 — 换成你们云效项目正式名。
  3. module-role-mapping.md — 换成你们模块 ↔ 责任部门对照。
  4. 成稿版式 — 若对外公告格式不同，改「固定输出格式」一节。
  5. 取数方式（references/yunxiao-sprint.md）— 浏览器会话或本机 OpenAPI 环境变量；禁止把 Token 写进仓库。

建议分享包最小内容

AutoVUL/ SKILL.md examples.md input-template.md module-role-mapping.md references/ yunxiao-sprint.md

7. 边界说明（跟同事对齐用）

类型 说明
会触发 说「AutoVUL / 版本更新日志 / 按迭代生成」；或给出迭代名称
只读云效 读迭代与关联需求；不创建【发版】任务、不改状态（发版挂链仍用云效生命周期 Skill）
可跳过 / 兜底 无云效权限时用手动清单；纯技术债条目不进对外日志
不会 电脑无人操作时后台自动发公告；也不替代人工确认发版窗口
