# 云效需求描述：MD 全文直写（强制）

发云效 / 完善需求描述时，**禁止**只贴摘要、口头复述或「浓缩交付口径」代替全文。  
必须把本地 PRD **Markdown 文件正文原样写入**云效需求「描述」字段（保留 Markdown 标题、表格、列表、mermaid 源码）。

## 真相源文件（按优先级取第一个存在的）

1. `src/prototypes/<prototype-id>/.spec/requirements-prd.md`（主真相）
2. 若无主文件：`src/resources/prd/<prototype-id>-autoprd.md`

先读文件 → 再写入云效；不要凭记忆重写一版。

## 描述区拼装顺序（整段写入同一描述框）

```markdown
## 原型链接
<对象存储或预览 URL；没有则写「待发布」或本轮发布后回填>

## 需求说明
<requirements-prd.md 的完整 Markdown 正文，一字不改地粘贴>

## 更新内容
<第 10 章「功能变更记录」中自上次定稿以来的最新一块；无则「首版定稿」或「本轮无功能/逻辑增量」>

## 更新内容·历史
<云效里原有的「更新内容」全文倒序追加；勿删除历史>
```

说明：

- **需求说明** = MD 文件**全文**（含第 1–9 章；若文件已含第 10 章，全文仍整份粘贴，更新内容区可再摘最新定稿块便于扫读，允许与全文第 10 章重复）。
- 不要把 PRD 改写成纯文本段落；云效支持 Markdown / 富文本时优先 **Markdown 原文**。
- 若编辑器会吃掉 `#` / 表格：尽量用「Markdown / 源码」模式粘贴；失败则在回报中说明，并保留本地 MD 路径给用户手动粘贴。

## 执行步骤

1. 确认 `<prototype-id>` 与 MD 路径。  
2. `Read` 该 `.md` 全文。  
3. 组装上方四段（原型链接 + 全文需求说明 + 更新内容 + 历史）。  
4. 写入云效需求描述（覆盖「需求说明」区时用新全文；历史更新内容先挪走再写）。  
5. 回报：需求编号、MD 路径、描述是否全文写入成功、字数/章节数。

## 禁止

- 只用第 9 章「交付口径」或几句摘要顶替「需求说明」
- 二次改写、删减章节、去掉 mermaid / 表格
- 只写「详见原型 / 详见附件」而不贴 MD 正文（除非用户明确只要链接）
