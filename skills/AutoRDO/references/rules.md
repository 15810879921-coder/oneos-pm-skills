# AutoRDO 清洗与拆解细则

## 目标

碎片 → 可入库的书面「标题」与「描述」。保留原意；不写成产品 PRD。

## 必做与精炼规则

| 规则 | 说明 |
|---|---|
| **标题精确拆解** | 固定格式为：`[模块名]模块[核心功能]功能`<br>• **模块名精炼**：精准识别 2-6 字的标准业务/系统模块词（如：`验车管理`、`合同管理`、`车辆管理`），严禁包含长句或口语。<br>• **功能名精炼**：用 10-20 字概括“核心对象/触发条件 + 核心动作 + 业务约束”（如：`合同自动生成验车记录及入库控制`），彻底剔除“我们要做一个”、“主要是用于”、“完成之后才能”等废话。<br>• **合成示例**：`验车管理模块合同自动生成验车记录及入库控制功能` |
| **描述转译** | 原文的转译结果，将口述/碎片材料转译为规范书面表达，保留原意（不改变谁要求什么、约束与例外；不替换成「更优方案」） |
| 书面化 | 口语改成完整句子或清晰短条目；主谓齐全 |
| 去口头禅 | 删除：嗯、啊、那个、就是说、然后呢、对对对、你懂吧 等 |
| 自我修正 | 以最后一次明确口径为准；若前后矛盾且未决，列入「待确认」 |
| 去重复 | 同一意思多轮确认只保留一处 |
| 轻度格式 | 可用短标题/条目；禁止展开成总览/角色/流程图/故事点等 PRD 结构 |
| 去结尾句号 | 描述中每个句子或条目末尾去掉 `。`；保留 `？` `！` 若确为疑问/感叹 |
| 待确认 | 材料含糊处写 `待确认：…`，不臆造 |

## 标题提炼反例与正例对比

- ❌ **坏标题（直抄长句/口语）**：`验车管理模块我们现在要做一个采购合同和三方租赁合同自动形成验车记录验车完成之后才能车辆入库功能`
- ✅ **好标题（精准模块+核心动作）**：`验车管理模块采购及租赁合同自动生成验车记录与入库校验功能`

## 不做

- 不生成 AutoPRD 十章或「产品说明」
- 不写验收清单、故事点、mermaid
- 不改云效、不建【交付】/分析/设计
- 不加载 `yunxiao-requirement-lifecycle`

## 质量自检

- [ ] **标题格式严格为** `[模块名]模块[核心功能]功能`（模块词精炼无口语，功能名高度概括动宾语义）  
- [ ] **描述为原文的转译结果**，干系人能认出这是自己说的意思  
- [ ] 无口头禅堆砌、无未决矛盾被写成定论  
- [ ] 描述末尾无 `。`  
- [ ] 未冒充完整 PRD
